In this guide, you\u2019ll learn how to get started with Defend. Defend is a powerful cybersecurity tool that safeguards your organization against sophisticated inbound threats. Defend analyzes your mail flow before delivery and provides protection that keeps your users and sensitive data secure. You can use the Defend console to customize security settings and conduct thorough investigations of detected and user-reported phishing emails.
\n
To access the Defend console, you must first use the deployment center to deploy Defend for your organization.
\n
  
\n
Deployment Center
\n
The Defend deployment center is a comprehensive wizard that allows admins to configure and deploy Defend features to their organization easily. Admins are guided through each section of the deployment process, and progress is saved at every step. Once the deployment center is complete, admins gain access to the Defend console, where further customization can be completed.
\n
Prerequisites
\n
To successfully complete the deployment center process, make sure you have the following prerequisites:
\n
    \n
  • Microsoft 365 tenant
    • \n
  • Global admin role on Microsoft 365 tenant
    • \n
\n
Deployment Center Steps
\n
The following steps are completed in the deployment center:
\n
    \n
  1. \nValidation: Ensure your current email system is ready for deployment. Defend will analyze your email system to ensure compatibility with the deployment process and the Defend service.\n
     
    \n
    2. \n
  2. \nTenancy and Secure Email Gateway (SEG): Confirm your primary domain, industry, and SEG provider. For more information about the fields on the Tenancy and Secure Email Gateway (SEG) page, see the list below.\n
     
    \n
      \n
    1. \nPrimary Domain: Defend will detect and provide a list of internal domains. Review the list and select your organization's primary domain.
      2. \n
    2. \nIndustry: Use the drop-down menu to select the most appropriate industry for your organization. Industry information enables Defend to address industry-specific threats.
      3. \n
    3. \nSecure Email Gateway (SEG): Select if your organization uses a SEG.
      4. \n
    \n
    3. \n
  3. \nImpersonation: Customize your impersonation protection by providing your organization's associated domains and brand names. For more information about the fields on the Impersonation page, see the list below.\n
     
    \n
      \n
    1. \nSecondary Domains: Knowing your secondary domains allows Defend to extend protection to all your organization's domains and prevents impersonation attacks of these domains.
      2. \n
    2. \nBrand Names: Understanding your brand names enables Defend to detect and block brand impersonation attacks that attempt to use variations of your organization's trusted identity. Adding as many brand names as possible is key to maximizing protection against these impersonation attacks.
      3. \n
    \n
    \n
    \nWarning: Leaving the Brand Names field empty exposes your organization to an increased risk of impersonation attacks.
    \n
    \n
    4. \n
  4. \nLinguistic Analysis: Avoid false positives by listing custom subject line phrases used widely in your organization. For more information about the fields on the Linguistic Analysis page, see the list below.\n
     
    \n
      \n
    1. \nCustom Subject Phrases: Understanding your organization's common email subject phrases enables Defend to better distinguish between legitimate communications and sophisticated phishing attempts that mimic your organization's standard messaging patterns.
      2. \n
    \n
    5. \n
  5. \nDefend Features: Grant permissions for the User Analysis and Admin Features app registrations that allow Defend features to protect your users. For more information about the fields on the Defend Features page, see the list below.\n
     
    \n
      \n
    1. \nUser Analysis: Grant the User Analysis permissions so that Defend can analyze historical data to enhance threat detection and protection.
      2. \n
    2. \nAdmin Features: Grant the Admin Features permissions so that Defend can facilitate the viewing and remediation of emails from the Defend console.
      3. \n
    \n
    6. \n
  6. \nAdmins: Specify the Global Admins that will be managing your Defend console.\n
      
    \n
    7. \n
  7. \nUser Groups: Create or specify the Microsoft 365 group that will be used to control which users in your organization will have Defend functionality. For testing purposes, create or specify a banner override group that will allow users to receive Defend banners. For more information about the fields on the User Groups page, see the list below.\n
     
    \n
      \n
    1. \nUser Group Name: Specify the name of the group that will contain all the users that Defend will protect.
      2. \n
    2. \nOverride Group Name: Specify the name of the group that can be used to test Defend functionality while Defend is in silent mode for the rest of the organization.
      3. \n
    \n
    8. \n
  8. \nDeployment Summary: Review and edit the information you have provided.\n
      
    \n
    9. \n
  9. \nHealth Check: Add a test user to the Defend user group to run a health check against the user, ensuring successful deployment.\n
      
    \n
    10. \n
\n
Once deployment is complete, you can roll out Defend features to your organization by adding the required users to the Defend user group in Microsoft 365.
\n
Summarized Changes
\n
Once successfully deployed, the following changes will be made to your Microsoft tenant, allowing Defend features to function correctly.
\n
\n
Defend automatically adds an accepted domain to your Microsoft 365 tenant. Each Defend tenant has a unique domain assigned to it that is randomly generated.\n
For example, \u201caaaaa.c.us1.defend.com\u201d, where \u201caaaaa\u201d is a randomly generated string that is unique to each Defend customer.
\n
This feature allows:
\n
    \n
  • The Microsoft 365 inbound connector validates emails coming from Defend and attributes those emails to your Microsoft 365 tenant. The certificate used to send emails from Defend to your Microsoft 365 must match an accepted domain on your tenant to ensure uninterrupted mail flow.
    • \n
  • Defend to DKIM-sign all emails before returning them to your Microsoft 365 tenant. For security reasons, this signature is unique to each tenant and uses your unique domain.
    • \n

Defend User Group
\n
The default name for the Defend user group is \"Defend_User_Group\". Users in this group will have their emails processed by Defend.
\n
You can use dynamic expressions to add all users to the \"Defend_User_Group\" automatically. Two examples of dynamic expressions are:
\n
    \n
  • (user.mail -ne null)
    • \n
  • (user.userType -eq \"Member\").
    • \n
\n
For new deployments via the Deployment Center, the Defend User Group now includes the legacy graymail and spam groups mentioned below.
\n
Legacy Groups
\n
The following groups may exist in your Microsoft tenant if you have used a previous Defend deployment tool:
\n
    \n
  • \nGraymail User Group: The graymail group contains users who have access to Defend\u2019s graymail functionality. Custom group names can be specified in the Graymail User Group field on the Defend Settings page. For more information on managing and enabling the Graymail user groups, see the Defend - Graymail Management article.
    • \n
  • \nSpam User Group: The spam group contains users who receive spam protection functionality in Defend. Custom group names can be specified in the Spam User Group field. For more information, see the Defend - Spam Management article.
    • \n
\n
These legacy groups are still valid and can be used as normal.
\n
POV Groups
\n
For POV deployments, two additional groups are available as follows:
\n
    \n
  • \nDefend_Banners_Override Group: Allows admins to add specific users who will receive banners even when Defend is in silent mode. This feature enables IT admins to test Defend functionality and train users at their own pace.
    • \n
  • \nDefend_Graph_Users Group: A Graph API only group used during POV deployments where the SMTP transport rules are not activated. All Defend functionality operates via the Microsoft Graph API for users in this group.
    • \n

Defend Console
\n
Once deployment is complete and you have rolled out Defend to your organization by adding the required users to the Defend user group in Microsoft 365, you will have access to the Defend console.
\n
  
\n
\n
\nNote:By selecting the navigation menu at the top-left corner of the page, you can navigate to other KnowBe4 products, including Defend, PhishER, and KSAT. For more information, see our Login Page and Navigation Menu Overview article.
\n
\n
The console is comprised of the following pages:
\n
    \n
  • Dashboard\n
      \n
    • The dashboard provides an overview of the email activity observed by Defend. The information on the dashboard is displayed in easy-to-read graphs and charts.
      • \n
    • For further information, see the Defend - Dashboard article.
      • \n
    \n
    • \n
  • Recent Emails\n
      \n
    • The recent emails page allows admins to view and analyze emails processed by Defend.
      • \n
    • For further information, see the Defend - Recent Emails article.
      • \n
    \n
    • \n
  • Abuse Mailbox\n
      \n
    • The abuse mailbox page displays all the emails submitted to the abuse mailbox and their analysis status.
      • \n
    • For further information, see the Defend - Abuse Mailbox Automation article.
      • \n
    \n
    • \n
  • Allow or Deny List\n
      \n
    • These lists can specify what happens when a specified email address, domain, or IP address sends an email to a user in your organization.
      • \n
    • For further information, see the Defend - Allow or Deny Lists article.
      • \n
    \n
    • \n
  • Event Notifications\n
      \n
    • Event notifications can be used to trigger admin notifications when specific actions occur.
      • \n
    • For further information, see the Defend - Event Notifications article.
      • \n
    \n
    • \n
  • Settings\n
      \n
    • The settings page allows admins to customize Defend settings to suit an organization's needs.
      • \n
    • For further information, see the Defend - Settings article.
      • \n
    \n
    • \n
  • URL Rewriting or Decoding\n
      \n
    • Add URL exceptions that Defend will no longer rewrite. Use the decode tool to view Defend\u2019s rewritten URLs in their original form.
      • \n
    • For further information, see the Defend - URL Rewriting Exceptions and Defend - URL Decoding articles.
      • \n
    \n
    • \n
  • User Management\n
      \n
    • The user management page allows you to add, edit, and remove Defend admins.
      • \n
    • For further information, see the Defend - User Management article.
      • \n
    \n
    • \n
  • Audit Log\n
      \n
    • View changes in your Defend console made by all admins.
      • \n
