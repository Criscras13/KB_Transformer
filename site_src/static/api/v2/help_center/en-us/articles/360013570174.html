The Password Exposure Test (PET) is a free tool that analyzes the passwords of the accounts in your Active Directory (AD) for numerous types of password vulnerabilities. Using this test will increase your organization's awareness by letting you know if you're susceptible to a password-related attack.
\n
The PET results will display which user accounts failed the test, and what vulnerability or vulnerabilities they failed with. This information can empower you to enhance your internal security measures by training your users about password safety, enhancing your password complexity rules, or taking other actions to bolster your cybersecurity posture.
\n
How Does the PET Work?
\n
The PET analyzes the following AD information in order to check for weak and vulnerable passwords:
\n
    \n
  • Your AD password table (containing hashed passwords)
    • \n
  • The encryption algorithm of your AD password table
    • \n
  • The primary domain found in your AD tree
    • \n
\n
The PET also uses your business email domain to assess the amount of data that is publicly available about your organization.
\n
It is important to note that this tool will never display or report the actual passwords of any user accounts in your AD. Passwords within AD are in a hashed format and will not be visible at any point. The test results will simply identify the user accounts that fail the test so you can decide how to remedy that.
\n
Additionally, the data pulled from AD is encrypted. The information obtained during the test is saved in local memory, not to a disk. Your domains are the only data transmitted to us from your AD at any point during the test.
\n
System Requirements/Prerequisites
\n
System Requirements
\n
To run the PET, the system you use must have the following:
\n
    \n
  • Windows 10 or later (32 or 64-bit), Windows Server 2016 or later
    • \n
  • Active Directory (AD), running on Windows Server 2008 R2 or later
    • \n
  • Ability to access the domain controller (DC)
    • \n
  • Internet access
    • \n
  • .NET Framework 4.7.2 (will be installed if needed)
    • \n
  • At least two processors
    • \n
  • At least 2GB of RAM
    • \n
  • At least 1GB of hard disk drive (HDD) space available on your system drive
    • \n
  • User Account Control (UAC) enabled
    • \n
\n
You should also run this test on a system\u00a0other\u00a0than your DC\u00a0as the scanning process can temporarily generate significant network traffic and CPU usage.
\n
Prerequisites
\n
For installation, you will need the following information:
\n
    \n
  1. A license key, emailed to you when you sign up for PET
    2. \n
  2. Domain name of your AD (For example: MyDomain.com or MyDomain.local)
    3. \n
  3. Name of your Domain Controller (DC)
    4. \n
  4. Credentials to connect to your AD.
    5. \n
\n
\n
\nImportant: The credentials you use to connect to Active Directory with the PET must\u00a0have\u201cReplicating Directory Changes\u201d and \u201cReplicating Directory Changes All\u201d permissions for the test to run successfully. These permissions allow you to obtain a copy of your password table for analysis. For information on quickly adding required permissions to an account in AD, see our How to Grant \"Replicating Directory Changes\" Permissions article. For additional notes on these permissions, see the list below.\n
    \n
  • A domain admin\u00a0does not\u00a0have permission by default to access this information, so using the tool with a domain admin account will not necessarily allow you to run the test successfully.
    • \n
  • We strongly recommend creating a\u00a0new account\u00a0in AD with these permissions for the purpose of running this test. Once the test is complete, you should delete this new account in accordance with the principle of least privilege.
    • \n
  • \nWhy create a new account? Creating a new account will make it easier to determine when this test took place and which account accessed the information, should you need to look for that information in the future. It also makes it easier to remove those permissions: once the test is done, simply delete the newly-added user account.
    • \n

Analyzing Your Results
\n
Your PET results display the total number of AD accounts checked and specify the number of accounts that are considered vulnerable or not vulnerable.\u00a0
\n
Each of your AD accounts will be listed in the results table and a checkmark will indicate the specific vulnerability or vulnerabilities found on that particular account. You can also search for a specific account by entering characters into the search box.\u00a0
\n
The test also performs a data exposure assessment and calculates an Exposure Level percentage and Risk Distribution pie chart for your organization.
\n
Refer to the details below to learn more about the different types of results available in your PET.
\n
    \n
  1. \nVulnerable Accounts: Lists the total number of accounts that were assessed and found to have vulnerabilities.
    2. \n
  2. \nNot Vulnerable: The number of accounts that were assessed and determined to not have vulnerabilities.
    3. \n
  3. \nAll Accounts: The total number of accounts assessed during your test.
    4. \n
  4. \nFound Vulnerabilities pie chart: Compares all of the password vulnerabilities found in your AD and indicates which vulnerabilities are most prevalent in your organization.
    5. \n
  5. \nExposure Level:\u00a0The percentage of your employees that were found to have publicly exposed data. Any account meeting at least one of the following failure types (found in the PET results table) contributes toward this percentage:\u00a0Sensitive Info Exposed, Password Info Exposed, or Breached Password.For more information on these failure types, please see the Additional Results section below.\n
    6. \n
  6. \nRisk Distribution pie chart: A visual representation of the severity of the data that is publicly-available about your organization. Hover over the pie chart to identify the two segments or ranks of data exposure results, outlined below:\n
      \n
    • \nHigh Risk:\u00a0Results were found in publicly-available breach data, which could contain sensitive personal information about your employees. This information can be used to create a sophisticated social engineering attack against individuals or an organization.\u00a0
      • \n
    • \nVery High Risk: Results were found in publicly-available breach data, where passwords were leaked during the breach. These password breaches included cleartext passwords or password hashes.
      • \n
    \n
    7. \n
  7. \nRerun Test:\u00a0You can click this button to rerun the PET.
    8. \n
  8. \nSearch:\u00a0Use this search bar to find specific accounts in your results.
    9. \n
  9. \nExport to Excel or Export to PDF:\u00a0Use these options to export your PET results.
    10. \n
  10. \nCheckmark icon: A checkmark icon indicates the type of vulnerability found on the account.
    11. \n
  11. \nFailure Types/Vulnerabilities Found: You can click each of the failure types in the panel on the left-hand side to filter the results to only show the accounts having that vulnerability. See the next section for details on these failure types.\n
    12. \n

Failure Types/Vulnerabilities Found
\n
The PET analyzes your data to look for 12 different failure types that can leave your organization vulnerable to an attack. The following table explains each of the primary PET vulnerabilities included in your results.\u00a0
\n
    \n
  1. \nWeak Password:\u00a0This failure indicates that the affected account's password matched one of those listed in our Weak Password dictionary. These passwords are either very common, easy to guess, or have been made available to attackers because of past data breaches.
    2. \n
  2. \nBreached Password: This failure indicated that at the time of a data breach, the affected account's password matches a password that was associated with an email address at your domain. Your AD accounts are checked against a database of breached data that contains over one billion leaked passwords.
    3. \n
  3. \nPassword Info Exposed: This failure indicates that the affected account has been found in publicly available breaches that contain either cleartext passwords or password hashes. The results for this failure type are found in the fourteenth column in the results table.
    4. \n
  4. \nShared Password: This failure indicates that the affected account shares a password with at least one other account.
    5. \n
  5. \nEmpty Password: This failure includes accounts that do not have a set password.
    6. \n
  6. \nClear Text Password: This failure includes passwords that are stored in clear text in an Active Directory. This means the users' AD passwords are stored using reversible encryption.
    7. \n
  7. \nPassword Not Required: This failure includes accounts that have the capability of not having a password.
    8. \n
  8. \nPassword Never Expires: This failure indicates that the account has its password timeout set to zero. Because of this setting, even if the Password never expires check box in the user\u2019s properties is unchecked, their password will never expire. The PET will check password expiration settings in your organization\u2019s domain policies, fine-grained password policies, and user properties.
    9. \n
  9. \nLM Hash Password: This failure indicates that the affected account uses a Local Area Network (LAN) manager hash, which is an antiquated method. These passwords are vulnerable to brute force attacks and can be cracked in seconds.
    10. \n
  10. \nAES Encryption Not Set: This failure indicates that the account doesn\u2019t use Advanced Encryption Standard (AES) to encrypt the user\u2019s password. AES encrypts passwords with a 128-bit or 256-bit key. So, passwords that use AES encryption are less vulnerable to attacks.
    11. \n
  11. \nDES-Only Encryption: This failure indicates that affected accounts were set up using the retired Data Encryption Standard (DES) mechanism. This could be a result of old software that doesn\u2019t know how to react to AES.
    12. \n
  12. \nMissing Pre-Authentication: This failure indicates that affected accounts have an important security mechanism turned off. Without these security mechanisms, the accounts may be at risk of brute force attacks. These attacks can occur offline and are difficult to detect. The security mechanism, when enabled, creates an encrypted authentication request so that attempts to authenticate to the account are logged.
    13. \n

Additional Results
\n
Your PET results will display information on additional results beyond just the failure types. These additional details and failure types can help to determine which of your AD accounts are the most vulnerable, and therefore in need of the most immediate attention. See the list below for details.
\n
    \n
  • \nDomain: Signifies which Domain is used in the corresponding AD account.
    • \n
  • \nSensitive Info Exposed: The affected accounts were found to have some of their identity information publicly visible, leaving them susceptible to phishing attacks.
    • \n
  • \nSocial Info Exposed: The affected account was found to have a public social media presence.
    • \n
  • \nPassword Last Modified: The date the password was last changed for the corresponding AD account.
    • \n
  • \nBreach Entry Timestamp: The approximate date that the affected account was last involved in a breach.Please see below\u00a0for additional information about where we obtain our breach data.
    • \n
  • \nBreach Info: Describes the type of breach the account was involved in (i.e., how the breach occurred, and what type of sensitive data was leaked as a result (if this information is available)).\u00a0Please see below for additional information about where we obtain our breach data.

Frequently Asked Questions (FAQs)
\n
    \n
  • \nI had several users fail the test. What do I do now?First and foremost, have the users with vulnerable accounts change their passwords immediately.Secondly, train your users on proper password practices with security awareness training and remind them often. It is important for them to know that hackers can crack a password within seconds with the right tools in hand.KnowBe4 offers several courses covering proper password practices, which can be used to train your users.For many of the vulnerabilities, you\u2019ll also want to enforce stricter password requirements in your organization. We strongly recommend increasing your password complexity requirements and setting a rule to ensure passwords expire on a regular basis.\u00a0While we cannot advise you on the specifics of how to remedy all of the password vulnerabilities in your organization, we can point you in the direction of some great resources that can help.\n
      \n
    • TechNet:\u00a0Configuring Password Policies\n
      • \n
    • TechNet:\u00a0Best Practices for Enforcing Password Policies\u00a0
      • \n
    • Microsoft:\u00a0Password Guidance (Downloadable PDF)
      • \n
    \n
    • \n
  • \nCan I see what the weak and breached passwords are? No. The passwords are hashed and cannot be displayed.
    • \n
  • \nCan I run this test if I'm using Azure AD? No. This tool will only work with a local AD.
    • \n
  • \nCan I run this test multiple times?Absolutely. If you want to run the test again, click Rerun Test below the Risk Distribution pie chart. Be sure to download a PDF or Excel sheet of your current results before running a new test.
    • \n
  • \nWhat is the Weak Passwords dictionary and where can I find it?The Weak Passwords dictionary contains over 11 million weak and compromised passwords from past data breaches. Hackers use similar dictionaries to attempt to crack your organization's passwords.To find the Weak Passwords dictionary, navigate to the Password Exposure Test installation folder by following this path: \u201cC:\\ProgramFiles (x86)\\KnowBe4\\Password Exposure Test\u201d. Then, open the data.bin file to view the Weak Passwords dictionary.
    • \n
  • \nWhere do you get your breach data from? Can I see it?The breached data shown in the PET is obtained through researching publicly-available breach information. For privacy and security purposes, the PET database is proprietary information. Additionally, KnowBe4 partners with Spycloud.com to search past breaches. Spycloud is a well-respected online resource which specializes in allowing users to search for their email address to see if their information has been made available in past data breaches.
    • \n
  • \nMy anti-virus flagged this as dangerous. Is it?No, it is not dangerous. The PET\u2019s behavior could mimic that of a password-cracking tool used by hackers, which is why your antivirus may have flagged it as potentially dangerous.
    • \n
  • \nAre any log files generated during the test?Yes. A log file is created when you run the PET. The file can be found under C:\\ProgramData\\KnowBe4\\Password Exposure Test\\DebugLogs.
    • \n
