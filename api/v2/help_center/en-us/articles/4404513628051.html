The improved spoof intelligence policy, now included in the Tenant Allow/Block List, looks at an email\u2019s From address to determine if spoof intelligence should be applied. This new feature improves whitelisting phishing security tests and prevents certain errors, such as the \u201cWe could not verify the identity of the sender\u201d error, from occurring. The information in this article was gathered from Microsoft\u2019s Manage your allows and blocks in the Tenant Allow/Block List article.
\n
There are three main methods of whitelisting a sender within the spoof intelligence policy. These methods include full infrastructure wild-carding, domain-specific whitelisting, or a sender-specific allow. The levels of focus for whitelisting spoofed templates are determined and set by the admin or organization.
\n
We recommend that you use the full infrastructure wild-carding method as this is the easiest method to whitelist all emails from KnowBe4.
\n
For more details on the syntax for the Tenant Allow/Block List for spoof intelligence, see Microsoft\u2019s Domain pair syntax for spoofed sender entries in the Tenant Allow/Block List article.
\n
Accessing the Spoofed Senders Tab
\n
In order to use the spoof intelligence feature, you will need to access the Spoofed senders tab in Microsoft Defender. Follow the steps below to access the Spoofed senders tab.
\n
    \n
  1. Log in to your Microsoft 365 account and select Admin from the navigation pane. Alternatively, log in to your Microsoft 365 Defender portal.
    2. \n
  2. From the Microsoft 365 Admin Center, select the All admin centers tab under Admin centers.
    3. \n
  3. On the All admin centers page, click Security.
    4. \n
  4. In the Microsoft 365 Defender navigation pane, click the Policies & rules tab under Email collaboration.
    5. \n
  5. Select Threat policies.
    6. \n
  6. Click Tenant Allow/Block Lists.
    7. \n
  7. Click on the Spoofed senders tab.
    8. \n
  8. Click Add.
    9. \n

Frequently Asked Questions (FAQ)
\n
What is a PTR record?
\n
A PTR (or pointer) record is the domain that is found in a reverse DNS lookup of the source email server's IP address.
\n
Does spoof intelligence affect Direct Message Injection (DMI)?
\n
\n
\nNote: In February 2025, Microsoft is retiring the Application Impersonation admin role used by our Microsoft DMI connection. This change will affect your DMI connections and cause them to break. If you currently use DMI as your whitelisting method, we recommend you update your whitelisting to use advanced delivery policies.
\n
\n
DMI-delivered campaigns should not apply this policy. The DMI application is allowed to impersonate users within the tenant per the roles assigned. However, if a DMI message is still showing an \u201cUnverified Sender\u201d warning after being sent, this method may be a viable alternative to whitelist those senders from receiving the warning message.
\n
